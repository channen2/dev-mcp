# Capability Check

Before building or recommending new automation, verify whether an existing tool, prompt, or skill already satisfies the request. This repo (devflow-mcp) already has a growing set of GitHub/Jira/Slack tools, prompts that chain them into workflows, and Claude Code skills that trigger those workflows from natural language — it's easy to propose duplicate work without checking first.

## Steps

1. **Identify the concept.** Restate what the user is looking for in one sentence — the underlying capability (e.g. "notify a Slack channel when CI fails"), not their exact phrasing.
2. **Search tools.** Grep `src/tools.ts` for `registerTool(` blocks whose `title`/`description` relate to the concept. Prefer `ui/catalog.json` if it exists and is recent — it's generated by introspecting the live server (`npm run ui:build`), so it can't be stale or hand-edited wrong the way source comments can. Regenerate it if missing or if `src/tools.ts`/`src/prompts.ts` changed more recently than the file's `generatedAt`.
3. **Search prompts.** Grep `src/prompts.ts` for `registerPrompt(` blocks — these package multi-tool workflows behind a single parameterized call (e.g. `pr_review`).
4. **Search skills.** Grep the `description` frontmatter of every `.claude/skills/*/SKILL.md` — skills trigger on natural language and may already cover the workflow end-to-end, including edge-case judgment calls a bare tool/prompt doesn't encode.
5. **Judge the match:**
   - **Full match** — an existing tool, prompt, or skill already does this. Name it (file:line) and stop; don't build anything.
   - **Partial match** — existing tools could compose into this, but nothing packages them yet. Name which tools would compose, and note whether the missing piece is better as a new prompt (protocol-level, explicit args, usable by any MCP client) or a new skill (Claude-Code-specific, triggers on natural language, carries the judgment-call prose) — see `.claude/skills/pr-review/SKILL.md` alongside `src/prompts.ts`'s `pr_review` for a worked example of that pairing.
   - **No match** — nothing exists, and no combination of existing tools covers it. Say so plainly rather than assuming.

## Reporting back

Lead with the verdict (exists / partial / missing), then the evidence (specific file:line, tool name, or prompt/skill name). Only offer a build recommendation if the user asks for one, or if the verdict is "missing" and next steps are the obvious question.
